メモリーに入り切らない大量のデータから n 番目に大きい数値を探索させるにはどうするか？（Microsoft のコーディング面接）

Note

およそ 2018年に Careercup.com に CodeNinja さんに投稿された問題です。
分析

min-heap と max-heap の回答が寄せられてますね。

n 番目に大きい数値だと n 要素のデータ構造が最小のメモリー空間となります。

メモリーに入る切るデータが M 要素ならば、（それから処理のために必要なメモリーを差し引いたサイズが）一度に処理できる最大量となるはず・・・ですかね。

C++ であれば k + 1 要素の priority_queue に M 個のデータから順にインサートしていき k + 1 番目の要素をデリートしていく方式で良さそうです。

メモリーに入り切らない大量のデータから重複する数値を見つけるにはどうするか？ (Microsoft のコーディング面接)

分析

寄せられてる回答は重複値のアルゴリズムがないので、広がりがないですね…　質問者には具体的にイメージする重複するデータがあるでしょうから、誤解するまえに聞いた方が良いと思います。

かなりオープンな質問ですね・・・

  • データが数値なのか、画像なのか、文字列なのか

これだけでも謎ですが、単にメモリーに入り切らないにしても、データによっては圧縮するのが先かなと思います。

  • 圧縮アルゴリズム(Huffman/LZW)を使って良いのか

圧縮データのマッチングはうまくいけば早いですしね・・・

この手のオープンな質問はメモリーを節約する方式を考えるのがメインとなるでしょうけど・・・これは一筋縄じゃいかないです。

重複する数値を探す方式は色々ありますが、まずは単純な方式を考えると良いかと思います。

  1. 単純比較だと計算量は O(n^2) で補助空間は O(1) です。

  2. 同一値の集計を配列に保持させる方式だと計算量は O(n) で補助空間は O(n) です。

この計算は単に重複値を見つけて表示するだけなので、そのまま使うことができません。

メモリーが足りてないのだから O(n) の補助空間の時点でまずいんですが、分割してやれば個別にバッチ処理すれば問題ないです。

ビットベクターを使って重複データのマーキングをしておくのも良いアイデアですね。

正の整数 n に対し、最小の数 m を求める。m の桁を掛けると n　に等しくなるように計算するものとする。桁のオーバーフローは無いものとする。

これは桁を計算するのと m を計算する 2 段階で考えたいですね。

64 であれば 88 となります。これは 8 と 8 を掛けると 64 と等しいからです。

なぜ 4 と 16 や 2 と 32 では駄目かというと、オーバーフローしてるからです・・・

パフォーマンスは無視で良いと思いますよ・・・
